Whiteboard → iPad app 📲

The Shared Whiteboard now installs to the iPad Home Screen and runs full-screen — no address bar, no tabs. The goal was that you can't tell it's the browser. This is what changed and how to set it up.

tldraw PWA commit 852b248

What it does on iPad now

🖼️ Full-screen

Add to Home Screen → launches chromeless like a native app. A real app icon (the sage “boards” grid), not a Safari tab.

🤏 Real pinch-zoom

Safari's page-zoom and double-tap-zoom no longer fight the canvas — a pinch zooms the drawing, not the web page.

🚫 No bounce

Rubber-band scroll and pull-to-refresh are gone. The page can't scroll; the canvas owns every gesture.

👆 Finger-sized

The top toolbar's buttons grow to ~44 px on touch and sit clear of the status-bar safe area. tldraw's own UI already adapts.

The changes

All in one commit. Each file, and why it was needed.

web/public/manifest.webmanifest new

What

PWA manifest — display:standalone, name, icons, colors.

Why

Tells iOS to launch from the Home Screen chromeless instead of as a Safari tab.

web/public/icon.svg + 3 PNGs new

What

App icon: 2×2 “boards” grid, house sage/cream palette. SVG source rasterized to 180 / 192 / 512 PNG.

Why

Home Screen needs a real icon; iOS won't use SVG for apple-touch-icon, so PNGs are required. 192/512 feed the manifest.

web/index.html — head meta

What

Manifest link, apple-mobile-web-app-capable, theme-color, apple-touch-icon, and viewport maximum-scale=1, user-scalable=no, viewport-fit=cover.

Why

Makes it installable + full-screen; stops Safari zoom from fighting the canvas; draws under the status-bar safe area.

web/index.html — css page lock

What

overscroll-behavior:none, fixed body, no tap flash, plus a .wb-chrome block for safe-area padding + coarse-pointer tap targets.

Why

Kills bounce / pull-to-refresh / page-scroll and makes the toolbar finger-friendly on iPad only. Scoped so the tldraw canvas is untouched.

web/src/App.tsx

What

Added className="wb-chrome" to the toolbar and wb-chip to the id chip.

Why

Hooks for the CSS above — inline styles can't do @media / env(). No logic changed.

server.js

What

Added .webmanifest → application/manifest+json to the MIME map.

Why

The static server had no MIME for it; iOS ignores a manifest served as the wrong type.

whiteboard.service

What

WB_HOST: 127.0.0.1 → 0.0.0.0 (+ comment).

Why

The enabler — bound to localhost, the iPad literally can't reach it. 0.0.0.0 exposes it on the LAN.

README.md

What

New “On an iPad” install section; fixed the stale bind note.

Why

Document the deploy + install flow and the security tradeoff.

Not committed: web/dist/ — rebuilt, but gitignored (build artifact). The running server serves it from disk already.

Set it up on the iPad

  1. Build the UI (once, and after any web/ change): 
    cd "~/ClaudeChats/Shared Whiteboard/web" && npm run build
  2. Serve on the LAN. Stop the current server, then: 
    cd "~/ClaudeChats/Shared Whiteboard"
WB_HOST=0.0.0.0 node server.js
    (The bundled whiteboard.service already binds 0.0.0.0 if you run it as a service.)
  3. Open on the iPad. In Safari go to http://10.0.2.52:5858. (hostname -I prints the host's IPs.)
  4. Install. Share button → Add to Home Screen → Add. Launch it from the new “Whiteboard” icon — it opens full-screen.
⚠️ Security: binding 0.0.0.0 exposes the whiteboard to every device on the LAN, with no authentication — anyone on the network can read and edit boards. That's the intent for iPad-on-desk use. To scope it tighter, bind a specific interface (e.g. your Tailscale address 100.102.199.37) instead of 0.0.0.0.

Keep the iPad and host on the same network — the page and its WebSocket both talk to <the-host-you-opened>:5858, so LAN over plain http/ws just works.
